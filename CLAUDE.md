## Overview

Monorepo for a job scraping and matching platform. The backend scrapes job listings, matches them to users via fuzzy string matching, and sends WhatsApp notifications. The frontend is a Next.js app that consumes the backend API.

## Commands

### Infrastructure (Docker Compose)
```bash
docker compose up -d        # Start all services in background
docker compose down         # Stop all services
docker compose build --no-cache  # Rebuild images
docker compose logs -f      # Follow logs
docker compose ps           # Check container status
```

### Makefile
```bash
make migrate                          # Apply all pending migrations
make migrate-create MSG="description" # Create a new autogenerated migration
```

### Tests (run inside `backend/`)
```bash
pytest tests/ -v                                 # Run all tests
pytest tests/test_auth.py -v                     # Run a single test file
pytest tests/test_auth.py::test_login -v         # Run a single test
pytest tests/ -v --cov=. --cov-report=term-missing  # With coverage
```

CI sets `DATABASE_URL=sqlite:///:memory:` and `SECRET_KEY=ci-secret-key-only-for-tests`. Tests use `TestClient` (synchronous) with a per-test SQLite DB and `reset_rate_limiter()` to clear SlowAPI state between tests.

### Sync
```bash
curl -X POST http://localhost:8000/sync-global        # Trigger job sync (respects cache)
curl -X POST http://localhost:8000/sync-global/force  # Force sync (bypasses cache)
```

### Frontend (run inside `frontend/`)
```bash
npm run dev      # Dev server
npm run build    # Production build
npm run lint     # ESLint
```

## Architecture

### Services (docker-compose.yml)
- **db** — PostgreSQL 18-alpine, database `vagas`, exposed on host port 5433
- **backend** — FastAPI on port 8000, uvicorn `--reload`
- **frontend** — Next.js on port 3000
- **evolution** — Evolution API v2.3.7 (WhatsApp gateway) on port 8080, backed by its own Postgres DB
- **evolution_redis** — Redis 7, used only by Evolution API
- **scheduler** — Alpine container that calls `POST /sync-global` every 6 hours via `sleep 21600` loop

### Backend (`backend/`)

Layered architecture: `routers/ → services/ → crud/ → database.py`

**Key flows:**
1. `POST /users` → creates user with hashed password, skills list, and match threshold; triggers matcher against existing jobs
2. `POST /sync-global` → background `run_scraping_task()` for 9 predefined Brazilian tech terms (`services/sync.py:GLOBAL_JOB_TERMS`)
3. `run_scraping_task()` (`worker.py`) → AdzunaScraper → upserts jobs by URL → runs matcher for all users → queues WhatsApp messages to `notification_outbox`
4. `outbox_worker.py` — async loop (30s polling) sends queued messages via Evolution API, retries up to 5 times

**Matching algorithm** (`services/matcher.py`): for each user skill, computes `thefuzz.fuzz.partial_ratio` against job title (weight ×2) and description; a skill matches if weighted score ≥ threshold. Recommendation created if % of matching skills ≥ `user.match_threshold` (default 70).

**Scraping cache** (`services/scraper.py`): checks `ScrapeHistory`; skips Adzuna API call if a term with ≥85% `token_sort_ratio` similarity was scraped within the last 6 hours.

**Auth** (`services/auth.py`): HS256 JWT with access tokens (30 min) returned in response body and refresh tokens (30 days) set as HttpOnly cookies. Login is rate-limited to 10/min; global limit is 60/min (SlowAPI).

**Models:** `User`, `Job` (unique by URL), `Recommendation` (status: pending/applied/rejected), `ScrapeHistory`, `NotificationOutbox` (status: pending/sent/failed, max 5 attempts)

**Scrapers** implement `BaseScraper` (`scrapers/base.py`). Only `AdzunaScraper` exists; register new ones in `worker.py`.

### Frontend (`frontend/`)

Next.js 15 App Router (`src/app/`). Uses MUI v7 + Emotion. Currently minimal — only scaffold exists. Backend URL from `NEXT_PUBLIC_API_URL`.

## Adding a New Scraper

1. Create `backend/scrapers/<name>.py` extending `BaseScraper` (implement `scrape(term: str) -> list[JobCreate]`)
2. Instantiate it in `backend/worker.py` alongside `AdzunaScraper`

## Workflow

- Após concluir qualquer alteração solicitada, crie o commit automaticamente sem precisar ser pedido.

## AI Implementation Guidelines

- Follow the layered architecture: `routers/ → services/ → crud/ → database.py`
- New endpoints go in `routers/`, business logic in `services/`, DB access in `crud/`
- New models/schemas must be consistent with `models.py` and `schemas/`
- Schema changes require an Alembic migration (`make migrate-create MSG="..."`)
- New scrapers must extend `BaseScraper` and be registered in `worker.py`
- Do not introduce new dependencies without justification
